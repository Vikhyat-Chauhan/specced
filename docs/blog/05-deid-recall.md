# De-id recall as a safety metric

Most ML metrics are symmetric: precision and recall are equally important, and F1 balances them. De-identification is different. The cost of a false positive (flagging "Boston" when it's a city, not a patient location) is low — an annotator reviews it, or downstream processing strips it. The cost of a false negative (missing a patient name, date of birth, or medical record number) is a HIPAA violation.

This asymmetry means **recall is the only de-id metric that matters for safety**. Specced enforces this with a hard gate: recall < 0.95 fails the case, regardless of how well the FHIR extraction went.

## What counts as PHI

Specced tracks HIPAA Safe Harbor categories:

| Type | Examples |
|---|---|
| `NAME` | Patient name, provider name |
| `DATE` | Visit dates, birth dates, dates of service |
| `AGE` | Age if ≥ 90 |
| `LOCATION` | Street address, city, state, zip |
| `ID` | MRN, SSN, device ID, IP address |
| `PHONE` | Phone and fax numbers |
| `EMAIL` | Email addresses |
| `ORG` | Hospital name, employer |
| `PROFESSION` | Occupation if linked to identity |
| `OTHER` | Any other unique identifier |

In the synthetic dataset, PHI is generated by Faker and embedded verbatim in the note. Gold PHI spans record the exact text, type, and character offsets.

## The scoring model

De-id scoring matches gold and predicted spans by `(type, normalized_text)` — not by exact character offset. A model that identifies "Elizabeth Brown" as a `NAME` at position 0 vs position 5 still gets credit (the prediction is correct, just offset differently).

```python
gold_spans = {(s.type, normalize(s.text)) for s in gold.phi_spans}
pred_spans = {(s.type, normalize(s.text)) for s in pred.phi_spans}
recall = len(gold_spans & pred_spans) / len(gold_spans)
```

Precision is computed but not gated — a model that over-flags (high recall, lower precision) is safer than one that under-flags. The penalty for over-flagging is operational, not regulatory.

## Why 0.95 and not 1.0

A threshold of 1.0 would reject any example where the model misses even one PHI span. In practice, some PHI spans are ambiguous (a date that's also a common reference, an age that's not linked to a specific patient). A threshold of 0.95 allows one miss in 20 spans — enough headroom for genuinely ambiguous cases while still enforcing meaningful coverage.

For a production system, the threshold should be determined by the risk tolerance of the deploying organization. 0.95 is the training default; a hospital might require 0.99.

## Results

On the 20 held-out cases:

| Variant | De-id recall |
|---|---|
| Base model | 0.850 |
| Fine-tuned | **1.000** |
| FT + Agent | **1.000** |

The base model's 0.850 recall (missing ~1 in 7 PHI spans) would fail in any real deployment. The fine-tuned model hits 1.0 on the held-out set. The 160 training examples are enough to internalize both the PHI types and the output format for phi_spans.

Note that the synthetic dataset generates only 3–5 PHI spans per note (name, date, age, MRN, phone). A real clinical note might have 15–20 PHI spans. Performance on real notes would require evaluation on real (de-identified) benchmarks like i2b2 2014.

## The over-flag philosophy

The CLAUDE.md hard rule for de-id:

> **Recall is safety-critical for de-id: when in doubt, over-flag PHI rather than miss it.**

This is encoded in the system prompt:

```
De-id recall is safety-critical: flag ALL HIPAA Safe Harbor PHI — when in doubt, over-flag.
```

An LLM that hedges ("this might be a date") should lean toward including it in phi_spans. The downstream human reviewer is cheaper than a HIPAA breach.

## Connection to training

The de-id gate in `filter.accept()` doesn't use the 0.95 threshold directly — it uses the overall eval score (which weights de-id recall at 0.2 in the aggregate). A training example with de-id recall = 0.0 would fail the overall eval gate anyway. This means the reject-sampler implicitly filters out learnable examples with poor de-id — which is the right behavior.
